# edit-snapshot

一个给 OpenClaw 文件修改流程加保护的 skill。

它的目标很直接。在任务准备修改文件时，先为当前工作区建立一个可靠的本地 Git 恢复点。任务结束后，如果产生了新的文件改动，就把修改结果也记录下来，并把最近几次提交记录和可用的回退方式展示给用户。

这个 skill 不是为了替代正常的版本管理流程，而是为了在模型执行写文件、批量替换、代码生成、配置调整这类操作时，尽量把“先留后手，再继续修改”变成默认动作。

## 核心能力

这个 skill 主要做三件事。

### 1. 在修改前创建保护快照

当任务会写文件时，先执行 `pre`。

它会确认当前目录是否已经是 Git 仓库，并在需要时补齐最基本的本地配置。随后判断当前工作区是否已经存在未提交改动。

- 如果当前仓库已经有一个干净的 `HEAD`，它会直接把这个 `HEAD` 当作修改前恢复点
- 如果当前工作区已经有未提交改动，它会先把现场保存成一个本地提交，避免后续写操作把已有改动也混在一起
- 如果仓库还没有首个提交，它会补一个初始化锚点，确保后续快照和回退都有落点

### 2. 在修改后记录结果快照

任务完成后执行 `post`。

它会检查这次操作是否真的带来了新的工作区变化。

- 如果有新改动，就创建一个 `guard(post)` 本地提交作为结果快照
- 如果没有新改动，就不创建无意义提交，而是明确报告这次没有新增快照

### 3. 把历史和回退方式告诉用户

任务结束后，可以再执行 `recent` 和 `rollback-help`。

这样用户能立刻知道。

- 修改前恢复点是什么
- 修改后是否生成了结果快照
- 最近几次提交记录是什么
- 想撤回本次改动时有哪些常见方式

这正是这个 skill 的使用重点。它不只是“顺手 commit 一次”，而是把**修改前保护、修改后留痕、结果对用户可见**串成一个完整动作。

## 适用场景

适合这些场景。

- 修改代码文件
- 修改配置文件
- 批量替换文本
- 重构
- 自动生成文件
- 运行会落盘的命令或脚本

不适合这些场景。

- 纯只读分析
- 纯搜索
- 纯解释
- 只查看差异但不落盘

## 工作方式

脚本入口在。

```bash
./skills/edit-snapshot/scripts/helper.sh
```

它支持四个子命令。

```bash
./skills/edit-snapshot/scripts/helper.sh pre "reason"
./skills/edit-snapshot/scripts/helper.sh post "reason"
./skills/edit-snapshot/scripts/helper.sh recent 5
./skills/edit-snapshot/scripts/helper.sh rollback-help
```

### `pre`

用于记录修改前恢复点。

它会完成下面这些动作。

- 检查当前目录是否是 Git 仓库，不是则初始化
- 检查当前仓库是否配置了 repo-local 的 `user.name` 和 `user.email`
- 判断当前工作区和暂存区是否有未提交改动
- 在必要时把当前现场保存为 `guard(pre)` 提交
- 把本次会话的 PRE 信息写到 `.git/.git-guard/last-session.env`

### `post`

用于记录修改后的结果状态。

它会完成下面这些动作。

- 读取前一步写入的 session 信息
- 检查当前工作区是否还有新的未提交变化
- 在必要时创建 `guard(post)` 提交
- 如果没有新变化，则只报告 `POST_NONE`

### `recent`

用于展示最近几次本地提交记录。

默认显示最近 5 条提交。

### `rollback-help`

用于输出这次会话常见的回退建议。

它会给出几类典型命令。

- 用 `git diff PRE..POST` 查看本次改动范围
- 用 `git revert <post_sha>` 做非破坏性撤销
- 用 `git reset --hard <pre_sha>` 直接回到修改前
- 用 `git reflog -n 10` 查看更多可恢复位置

它本身不会直接执行回退，只负责把信息明确告诉用户。

## 行为边界

这个 skill 的边界也很清楚。

它依赖的是**当前目录下的本地 Git 仓库**。如果目录原本不是 Git 仓库，它会在本地创建一个。它不会把这个 skill 扩展成远端同步、协作流转或历史整理工具。

默认情况下，它不会主动进行远端同步，也不会改写历史。它做的是本地安全快照和回退提示，而不是替代团队的完整 Git 工作流。

## 一个需要提前知道的行为

如果任务开始前仓库已经是脏的，也就是已经存在未提交改动，那么 `pre` 会把这些现有改动一起保存到 PRE 快照里。

这意味着 PRE 快照未必只包含“本次任务”的内容，也可能包含任务开始前就已经存在、但尚未提交的本地修改。

这是这版 skill 的有意设计。它优先保护现场，避免在继续写文件时把已有内容一起覆盖或冲散。

## 建议的用户回报格式

任务完成后，建议按下面这种方式告诉用户结果。

```text
已完成本次修改，并已创建本地 git 保护记录。

PRE 快照: <sha> <subject>
POST 快照: <sha or none> <subject or reason>
最近提交:
<recent output>

支持回退。
如需非破坏性回退，可用:
  git revert <post_sha>
如需直接回到本次修改前，可用:
  git reset --hard <pre_sha>
如需查看更多恢复点，可用:
  git reflog -n 10
```
